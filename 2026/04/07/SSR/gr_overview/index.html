生成式推荐的工作概览

参考博客：知乎州懂的博客

一、目前生成式推荐的发展路线

现阶段推荐系统的发展路线主要有两条：

  • DLRMs：传统判别式模型，为了实现推荐系统的Scaling Laws，最近的工作主要集中在LLM4DLRMs上，代表有 MTGR（美团）、RankMixer（字节）、OneTrans（字节） 等。
  • GRs：通常就是指LLM4GRs，偏激进的探索推荐系统的新范式，比如直接生成候选item、用户意图理解等，代表有 Tiger（Google）、OneRec/OneRec-Think（快手） 等。

二、LLM4DLRMs路线

传统的DLRMs（Deep Learning Recommender Models）主要是基于深度学习的判别式模型，近年来面临着增长见顶的困境，各种新模块的引入带来的边际收益逐渐变窄，对比生成式LLM传统DLRMs的不足可以概括为以下几点：

  • 稀疏ID表征的问题：传统的DLRMs为每个itemID分配一个hashID来得到物品ID的表征，HashID基本无冲突并且易学习，但是其局限性也很明显：
    • 缺乏多模态语义信息，冷启动问题突出：HashID本身没有懈怠任何多模态语义信息，导致严重的语义孤岛问题，同时新冷内容由于交互不足，HashID的表征质量很差，加剧了冷启动问题；
    • 参数庞大，难以学习更泛化的共现知识：ID表征没有语义共享，巨大的emb tabel占据了模型超过90%的参数量，计算和存储开销很庞大，同时由于ItemID的稀疏性，更高阶的（二阶/三阶）的item共现知识更难学习到。
  • 建模范式相对低效：传统的DLRMs普遍采用 Embedding layer + 序列建模 + 特征交叉 的建模范式，通常依赖于多种专用、异构的计算算子，造成通信开销大、硬件利用率的低下：
    • 缺乏统一的Token化处理：传统的DLRMs对于不同的特征类型（如ID、数值、文本等）采用不同的处理方式，缺乏统一的Token化处理，这些处理后的特征是各种大小不一的Embedding向量；
    • 计算算力碎片化：传统的DLRMs以稀疏查表为主，访存密集但计算稀疏，同时存在子模块（比如特征交叉、序列建模、多任务学习等），这些子模块的输入和计算方式不同意，会使得GPU算力非常碎片化；
    • 缺乏可预测的Scaling Laws：传统的DLRMs因为其输入特征异构、数据分布极度稀疏、模型结构简单并依赖人工特征工程，导致参数量和数据量的增加无法稳定、可控地提升模型性能。

2.1.如何缓解ID Embedding的问题

2.1.1.融合多模态Embedding

针对ID Embedding缺乏语义信息的缺点，近年来很多工作都提出使用多模态 Embedding 与 ID Embedding 做融合的思路, 包括 24 年闲鱼 MetaSplit, 蚂蚁的 ASIF, 以及前阵子阿里刚提出的 MUSE 方法, 其中 MUSE 方法成功将多模态用于 10W 级别的长序列建模中。MUSE 基于构造的I2I贡献数据，用对比学习微调得到多模态表征，然后GSU使用多模态的表征做Lifelong的TopK检索，之后在ESU用SA-TA注意力极致融合多模态信息。

具体来说，GSU阶段从长达10万级的用户历史行为中，用余弦相似度筛选出TopK个行为，得到一个相似度序列 R，然后将相似度范围 [−1, 1] 划分为 N 个区间，统计各个区间内的行为数量生成一个 N 维直方图 hMM；

而在ESU阶段的重点则是基于多模态相似度来改进DIN的注意力分数，其中 $\boldsymbol{\alpha}^{\text{ID}} = \frac{(\mathbf{E}_u W^k)(e_a W^q)^\top}{\sqrt{D}}$ 表示常规的DIN得到的注意力分数，结合GSU检索得到的相似度分数 R，将两者通过多个可学习参数融合作为最后的注意力分数：

$$ \begin{split} \boldsymbol{\alpha}^{\text{Fusion}} = \gamma_1 \boldsymbol{\alpha}^{\text{ID}} + \gamma_2 \mathbf{R} + \gamma_3 (\boldsymbol{\alpha}^{\text{ID}} \odot \mathbf{R}) \end{split} $$

之后再和 V 计算：ulID = Softmax(αFusion)⊤(EuWv)。

最终，将GSU和ESU两阶段的输出拼接起来送入后续的CTR预测模型：

$$ \begin{split} u_l = \text{Concat}(h^{MM}, u_l^{\text{ID}}) \end{split} $$

但是线上直接拉取多模态表征的开销还是很大的，使用量化后的多模态语义ID可以减少线上serving的难度，因此工业界最近一年的研究大多都在往多模态语义ID方向发展。

2.1.2.赋予Sparse Table语义结构

传统推荐系统通过Random Hash的方式讲一个Item ID映射到一个巨大的Embedding Table中，但是这样有两个比较大的问题：

  • 语义不相关的ID可能会撞到一起：比如猫咪和挖掘机的ID可能会被映射到同一个Hash ID上，这会导致模型训练的时候权重更新矛盾；
  • 新内容（冷内容）不友好：新上线的内容没有任何历史交互数据，哈希到随机行，完全从零开始学习。

针对这个问题，最近的工作提出了 语义ID 的概念，即通过某种方式将Item ID映射到一个具有语义结构的空间中，但是该怎么把这个语义ID变成embedding呢？这里Google和Meta分别提出了两种不同的思路。

Google的N-gram-based和SPM-based方法
  • N-gram-based方法：假设语义ID一共有 L 层，当 N = 2 时，如果空间不能重叠，这样就有 (c1v, c2v), (c3v, c4v), …, (cL − 1v, cLv) 共 L/2 个二元组，这里每个子词都会对应一个embedding table的某一行，最终的ID embedding就是这 L/2 个子词的embedding的平均值。
  • SPM-based方法：N-gram的效率是比较低的，因为这里预设的 N 是固定的，无法自动适配语义ID语料的不同分布。因此谷歌提出将SentencePiece用于处理语义ID的语料，具体来说，我们会将所有内容的语义ID串起来作为训练语料，然后BPE会不断合并出现频率最高的相邻pair，比如最开始是 c1v, c2v, c3v, c4v，如果 c2v, c3v 这个pair出现的频率最高，那么就将它们合并成一个新的子词 c23v，这样就可以得到一个更高效的子词表。因此最终学习到的分词方案是不固定的，而是根据语义ID的语料分布自动适配的，同时每个子词也会对应一个embedding table的行，最终的ID embedding就是这些子词的embedding的平均值。
Meta的前缀语义ID方法

Meta则提出了前缀语义ID的方法，具体来说，他们会将一个Item ID映射成一个前缀树上的路径，比如一个Item ID可能会被映射成 c1v → c2v → c3v 这样的路径，那么最终的ID embedding就是这个路径上所有节点的embedding的平均值。相比于Google的方法，Meta的方法更简单直接，同时也具有一定的语义结构，因为同一类item可能会共享相同的前缀。

以三层512的语义ID为例，ItemID的Sparse Table的大小就为 512 + 5122 + 5123 = 2.5M，假设一个Item的语义ID为 (123, 234, 345) ，那么它的ID embedding就是：SparseTable[123] + SparseTable[512 + 123 * 512 + 234] + SparseTable[512 + 5122 + 123 * 5122 + 234 * 512 + 345]。

2.1.3.使用语义id进行特征增强

快手DAS中提出了各种使用sid来为传统判别式推荐系统提供特征增强的方法：

  • ID-based特征：如果一个item的sid是[123, 234, 345]，那么就可以将对应的特征表示为 ad_l1=123，ad_l2=123_234，ad_l3=123_234_345；
  • List-wise ID-based特征：对于一个用户的历史行为序列，生成可学习的SID列表特征，比如用户点击了广告 [ad1, ad2]，对应的sid分别为 [76, 12, 67] 和 [76, 22, 242]，则生成的列表特征为 user_l1_sids=[76, 76]，user_l2_sids=[76_12, 76_22]，user_l3_sids=[76_12_67, 76_22_242]；
  • Cross ID-based特征：基于用户行为序列item的sid与候选item的sid之间的匹配计数，生成科学系的稀疏ID特征。比如用户历史列表为 [76_12_67, 76_22_242]，候选item的sid是 [76, 22, 131]，则生成的匹配计数特征为 cross_l1_count=1（因为用户历史列表中有一个item的l1 sid是76），cross_l2_count=1（因为用户历史列表中有一个item的l2 sid是76_22），cross_l3_count=0（因为用户历史列表中没有item的l3 sid是76_22_131）。

2.2.传统DLRMs如何实现Scaling Law

2.3.1.样本组织上

传统的DLRMs在样本组织上有两个比较大的问题：

  • PointWise样本组织格式效率低：传统的DLRMs按照<User, Item>的二元组来组织训练样本，对于同一个请求的m个item，会处理成m个样本，其中这些样本的用户侧特征和上下文特征完全相同，导致计算效率较低；
  • 不同长度序列的padding处理：在处理不同长度的序列数据时，往往用padding进行补全，但在超长行为序列建模时，zhexiepadding的token数量占比会非常多，拉低了模型的有效训练样本密度。
